# Lab Docs Engineering（科研文档工程化）

一个**可移植的通用 Skill**：本体安装一次（个人 `~/.cursor/skills/` 或某仓库 `.cursor/skills/`），
每个仓库用一份**项目内** `PROJECT_PROFILE.md` 特异化。无 profile 时先跑向导，再做业务动作。
安装与分发见 [README.md](README.md)。

## 北极星：为什么这能提升科研进度（也是自检项）

科研的真正瓶颈常是**连续性与交接**，不是算力：重复踩坑、baseline 混表、找不到当前主线、结案后证据丢失，才最拖进度。每次动作后，下面四条必须成立——冲突时**以这四条为准**，格式细则都为它们服务：

1. **决策+依据被固化**：结论写清 Go / No-Go 与**判据来源**，下次不重新 litigate、不重复踩坑。
2. **不跨路线混表**（正确性问题，不是格式）：不同 route 的指标不裸比；报指标必带 baseline / split / seed。
3. **可发现性**：新人从入口 README 几分钟能定位「当前主线 + 最新结论 + 下一步」。
4. **归档不丢证据**：收尾/归档后，现行文档仍保留 Go/No-Go 与证据路径（run 目录、图、summary）。

## 0. 每次必做：锁定 Profile

```
找 .cursor/skills/lab-docs-engineering/PROJECT_PROFILE.md ?
├─ 有  → 以它为准，直接做业务动作（§1 能力地图）
└─ 无  → 进向导 wizard.md，默认走「A 自动推断」：
         扫描仓库 → 草拟填好的 profile → 用户一键确认/微调 → 落盘 → 再继续
         · 仓库尚无 docs 结构 → 向导「B 从零 scaffold」先建最小骨架
         · 用户要完全手填     → 向导「C 完整问卷」
```

- 用户说「初始化实验记录 / 配置本仓库文档规范 / 重新配置」：即使已有 profile 也**重跑向导并覆盖**（先展示 diff 摘要）。
- 之后所有路径、分册、模板字段、归档策略 **以 profile 为准**；本 SKILL 只提供通用流程。
- Profile 写在**目标项目**内（可进 git），**不**写回 `~/.cursor/skills/`（那是通用本体，要保持项目无关）。

## 1. 能力地图（按用户意图分流）

| 意图 | 动作 | 详参 |
| --- | --- | --- |
| 记一下 / 写推进记录 / 代码改完了 | 文首倒序追加推进记录 | [templates.md](templates.md) §推进记录 |
| 正式入账 / 填实验表 / 对齐 xlsx | Markdown 日志 + 按规范填/对表 | [templates.md](templates.md) §正式入账 |
| 同步文档 / 整理知识库 / 收尾可接手 | 分层检查、去重、改入口 | [boundaries.md](boundaries.md) + profile 分层表 |
| 归档（仅用户明确说） | 移入 archive、留结论、修链接 | [archive-checklist.md](archive-checklist.md) |
| 分析后回填 | 只回填状态/指标摘要到日志与跟踪表 | 不替代分析 Skill，见 §6 |
| 改规范 / 演进口径 | 更新规范目录下 md；大改先口头确认一句 | profile.`norms_dir`，见 §8 |
| 初始化 / 配置 / 重新配置 | 跑向导生成/更新 profile | [wizard.md](wizard.md) |

可并行触发多条（例如：分析结论 → 推进记录 + 跟踪表 同时回填）。

## 2. 硬边界（默认禁止）

除非用户在当前对话**明确要求**，否则不要：

- 代跑训练、提交作业（Slurm 等）、批量评测
- 移动/删除 `outputs/`、`data*`、模型权重等大数据或证据目录
- 自动 `git commit` / `push`
- 写长篇实验复盘报告（那是分析类 Skill 的职责；本 Skill 只做**结构化回填**）

详见 [boundaries.md](boundaries.md)。

## 3. 默认执行强度

- **直接改 md**（推进记录、跟踪表、入口 README、规范文档），改完用 3–6 条 bullet 说明改了哪些文件与要点。
- **大改规范**（增删字段、改评估口径、改分册规则）：先用一句话确认用户意图，得到肯定再写。
- **归档**：必须用户明确说「归档」；不要仅因结案/No-Go 自动搬文件。

## 4. 推进记录工作流

1. 用 profile 的**路由表**选对日志文件（多路线时宁可先问一句，也不要混写）。
2. 新条目插入在「文件说明块之后、第一条旧记录之前」（**最新在上**）。
3. 标题：`## YYYY-MM-DD｜主题 · 动作 · 状态`（或 profile 覆盖格式）。
4. 状态用**绝对词**：`已提交` / `训练中` / `完训` / `No-Go` / `待验证` 等；禁止「今天 / 最近 / 昨天」。
5. 字段用统一模板（工程四件套 + 科研可选字段），见 [templates.md](templates.md)；profile 可勾选必填项。
6. 指标必带来源路径（run 目录、`summary.json`、图目录），且**不跨路线裸比**（北极星#2）。

完成实验分析或核心代码变更后：**主动**起草并写入推进记录（不必等用户再说「记一下」），仍遵守路由与边界。

## 5. 正式实验入账工作流

1. 确认 profile 的 `experiment_id_rule`、run 产物根、xlsx/CSV 路径。
2. 在路线日志/状态表写可追溯条目（exp_id、配置、split、关键指标、产物路径）。
3. 若启用 xlsx：按 `norms_dir` 内填写规范对齐列；缺文件则只维护 Markdown + index，并注明缺口。
4. **不编造**未跑实验的指标。

## 6. 知识库整理工作流

1. 读 profile 的文档分层表，区分：入口 / 总纲 / 路线状态 / 规范 / 推进记录 / 证据。
2. 事实写到最贴近事实的位置；入口只导航；推进记录只做时间倒序凭证。
3. 同一实验状态只在**一个**主文档维护，别处链接或一行摘要。
4. 清理相对时间词；修好断链。
5. 与项目内分析/同步类 Skill 的边界见 [boundaries.md](boundaries.md)，避免双写冲突。

分析衔接：分析 Skill 产出 Go/No-Go 与指标后，本 Skill 只**短回填**（推进记录一条 + 跟踪表顶块 + 待办行），不重做曲线对比、不凭记忆报数。

## 7. 归档工作流

仅当用户明确要求归档时，严格执行 [archive-checklist.md](archive-checklist.md)。核心：证据留在现行文档或 archive README，绝不随文档一起丢（北极星#4）。

## 8. 规范演进（本 Skill 可改规范）

- 小改（错字、链接、示例）：直接改 `norms_dir`。
- 大改（字段、口径、分册、入账流程）：先确认 → 改规范 → 同步改 `PROJECT_PROFILE.md` 对应键 → 推进记录记一笔。
- 通用科研**默认不绑定**具体路线名；分册规则由向导写入 profile。

## 9. 完成动作后的自检（30 秒）

对照北极星四条快速核对，任一不满足则补：

- [ ] 结论含 **Go/No-Go + 判据来源**（决策类动作）
- [ ] 指标带 **来源路径**，且**未跨路线裸比**
- [ ] 入口/状态文档能让新人定位「当前主线 + 最新结论」
- [ ] 无「今天/最近」等相对时间词；状态用绝对标签
- [ ] 归档动作后，Go/No-Go 与证据路径仍在现行文档可查
- [ ] 已用 3–6 条 bullet 汇报「改了哪些文件 + 要点」

## 10. 分发 / 向其他同学安装

见 [README.md](README.md)。要点：本体是**项目无关**的，装到个人 `~/.cursor/skills/` 后可跨自己所有仓库复用；每个新仓库首次对话说「初始化实验记录」自动进向导生成本仓 profile。
